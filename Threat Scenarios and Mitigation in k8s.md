    <article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-scope" class="anchor" aria-hidden="true" href="#scope"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Scope</h1>
<p>There are many dimensions to cover wrt securing the App and k8s cluster. The following article doesn't cover the following:</p>
<ul>
<li>Threat Detection</li>
<li>Build Hygiene</li>
<li>Image Hygiene</li>
<li>SecOps</li>
</ul>
<p>The best practice around k8s is to use <em>defense in depth</em> (layered defense) approach and redundancy. What is simply means to have layers of protection which makes it difficult for an attacker to get further even if they get thru a single layer (to prevent breach or limit it's impact).</p>
<h2><a id="user-content-scenario-1" class="anchor" aria-hidden="true" href="#scenario-1"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Scenario 1</h2>
<p>Attacker is able to access k8s API server.</p>
<h3><a id="user-content-details" class="anchor" aria-hidden="true" href="#details"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Details</h3>
<ul>
<li>Say due to a App side vulnerability an attacker gains access to Service Account associated with a POD and get access to the token which is stored on the POD.</li>
<li>Every SA has roles and permissions associated with it. Attacker can use the token to attack the cluster via requests to API server (in case the permissions are overly permissive and not constrained by RBAC).</li>
<li>Get access to secrets to further attacks</li>
</ul>
<h3><a id="user-content-mitigation-1-use-rbac-to-control-the-permissions-for-sas" class="anchor" aria-hidden="true" href="#mitigation-1-use-rbac-to-control-the-permissions-for-sas"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 1: Use RBAC to control the permissions for SAs</h3>
<ul>
<li>Use RBAC in k8s. Provide roles to SAs or users.</li>
<li>Implementation:
<ul>
<li>Create roles which are a collection of permissible actions (permissions) on the cluster.</li>
<li>RBAC setting can vary by namespaces or can apply to entire cluster.</li>
<li>tightly control the roles given to users and SAs</li>
</ul>
</li>
</ul>
<p>Permission--&gt; Verb + Resource type (for ex <em>get secrets, update configmap</em>)<br>
ClusterRole--&gt; Collection of permissions<br>
ClusterRoleBinding--&gt; used to connect a user or SA to a cluster role</p>
<p>So each user can have many roles associated with them. ClusterRole or ClusterRoleBinding can apply to a single namespace.</p>
<h3><a id="user-content-mitigation-2-restrict-access-to-api-server" class="anchor" aria-hidden="true" href="#mitigation-2-restrict-access-to-api-server"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 2: Restrict access to API server</h3>
<ul>
<li>In GKE use <em>Master Authorized Network</em> feature to restrict who can communicate to API server</li>
<li>Firewall API server via port or regular firewall rules to certain IP addresses (so an attacker cannot even send a request or not accessible to internet)</li>
</ul>
<h3><a id="user-content-mitigation-3-network-policy-based-mitigation" class="anchor" aria-hidden="true" href="#mitigation-3-network-policy-based-mitigation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 3: Network Policy based mitigation</h3>
<ul>
<li>Restrict access to DBs to only PODs which require it</li>
<li>Specify access using labels. Use <a href="https://docs.projectcalico.org/security/calico-network-policy" rel="nofollow">Calico</a>/ <a href="https://kubernetes.io/docs/tasks/administer-cluster/network-policy-provider/weave-network-policy/" rel="nofollow">Weave</a> to implement label (POD) based rules for accepting and sending requests</li>
</ul>
<h2><a id="user-content-scenario-2" class="anchor" aria-hidden="true" href="#scenario-2"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Scenario 2</h2>
<p>Attacker is able to access cluster components and manipulate them.</p>
<h3><a id="user-content-details-1" class="anchor" aria-hidden="true" href="#details-1"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Details</h3>
<p>Attacker can create PODs and read secrets or modify objects. Via API server access the attacker can spin more PODs and instances, read secrets etc.</p>
<h3><a id="user-content-mitigation-1-secure-etcd" class="anchor" aria-hidden="true" href="#mitigation-1-secure-etcd"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 1: Secure etcd</h3>
<ul>
<li>Use Authentication &amp; firewall to restrict access to etcd</li>
<li>Encrypt data in etcd (encryption @ rest)</li>
</ul>
<h2><a id="user-content-scenario-3" class="anchor" aria-hidden="true" href="#scenario-3"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Scenario 3</h2>
<p>Attacker is able to gain access to POD Host resulting in attacks to kubelet and other containers on the host.</p>
<h3><a id="user-content-details-2" class="anchor" aria-hidden="true" href="#details-2"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Details</h3>
<p>This is possible in scenarios where the App is running 3rd party application or accepts data or code from the user. This allows the attacker to upload and trick the App into executing a piece of malicious code. An Attacker can also gain access thru kernel or container vulnerabilities. Attacker could read files from the FS or execute application on host outside the container. This would allow them to see and read other containers running on the host and access their secrets associated with the containers. Once access to the host is gained, attacker can read the kubelet config (authentication and certs) and attack other containers on the same host, attack the cluster and get further escalating privileges from API server.</p>
<h3><a id="user-content-mitigation-1-run-apps-via-non-root-user" class="anchor" aria-hidden="true" href="#mitigation-1-run-apps-via-non-root-user"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 1: Run Apps via non root (user)</h3>
<ul>
<li>Run Apps in containers as user than root.</li>
<li>Make root FS read only. For use cases where the App needs to write some data create a temp directory or use an external volume. This prevents attackers from overwriting files from FS. So App bugs which allow file to be uploaded anywhere are mitigated.</li>
<li>Set <em>no_new_privs</em> flag: So if the App is executing a new binary then extra privileges are not granted.</li>
</ul>
<pre><code>apiversion: v1
kind: Pod
metadata: 
   name : somename
spec: 
  securityContext:
    runAsUser: 1000
    readOnlyRootFileSystem: true
    AllowPriviligeEscalation: false
</code></pre>
<h3><a id="user-content-mitigation-2-sandboxed-pods-for-less-trustworthy-pieces-of-applications-difficult-to-implement-generically" class="anchor" aria-hidden="true" href="#mitigation-2-sandboxed-pods-for-less-trustworthy-pieces-of-applications-difficult-to-implement-generically"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 2: Sandboxed PODs for less trustworthy pieces of applications (Difficult to implement generically)</h3>
<ul>
<li>Pods are sandboxed from other pods on the same host. Sandbox allows 2 layers of isolation (Linux Kernel and Sandbox)</li>
<li>Use <a href="https://katacontainers.io/" rel="nofollow">Kata containers</a> or <a href="https://gvisor.dev/" rel="nofollow">Gvisor</a> to have different runtimes for containers than the normal one. This allows an extra layer of isolation between container and host systems.</li>
</ul>
<h3><a id="user-content-mitigation-3-use-seccomp-or-apparmorselinux-to-create-layers-around-the-app" class="anchor" aria-hidden="true" href="#mitigation-3-use-seccomp-or-apparmorselinux-to-create-layers-around-the-app"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 3: Use <a href="https://en.wikipedia.org/wiki/Seccomp" rel="nofollow">seccomp</a> or <a href="https://wiki.ubuntu.com/AppArmor" rel="nofollow">AppArmor</a>/<a href="https://www.redhat.com/en/topics/linux/what-is-selinux" rel="nofollow">SELinux</a> to create layers around the App</h3>
<ul>
<li>App (Layer 0)</li>
<li>seccomp (Layer 1)</li>
</ul>
<pre><code>apiversion: v1
kind: Pod
metadata: 
   name : somename
   annotations: 
      seccomp.security.alpha.kubernetes.io/pod: runtime/default
...
</code></pre>
<ul>
<li>AppArmor/ SELinux (Layer 2)</li>
</ul>
<p>The only downside being that this needs intimate knowledge of how the App works.</p>
<h3><a id="user-content-mitigation-4-restrict-kubelet-permissions" class="anchor" aria-hidden="true" href="#mitigation-4-restrict-kubelet-permissions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 4: Restrict kubelet permissions</h3>
<ul>
<li>RBAC for kubelet- special RBAC for a Node. This restricts read access of secrets on cluster for nodes  (only to the ones secrets attached to the Pods on the node)</li>
</ul>
<pre><code>--authorization-mode= RBAC,Node
--admission-control=....,NodeRestriction
</code></pre>
<ul>
<li>Rotate certs - to restrict the period during which access exists.</li>
</ul>
<p>(GKE takes care of these by default)</p>
<h2><a id="user-content-scenario-4" class="anchor" aria-hidden="true" href="#scenario-4"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Scenario 4</h2>
<p>Unsecured PODs: Ensuring security standards are followed in the team while creating PODs.</p>
<h3><a id="user-content-details-3" class="anchor" aria-hidden="true" href="#details-3"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Details</h3>
<p>It is important to prevent creation of unsecured PODs even accidently by team members. Basically you need a policy mechanism to restrict provisioning of PODs which do not tick all the compliance boxes</p>
<h3><a id="user-content-mitigation-1" class="anchor" aria-hidden="true" href="#mitigation-1"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 1:</h3>
<ul>
<li>Use PodSecurityPolicy. If Pods created don't match the security policy the API server will reject them.</li>
<li>Use <a href="https://www.openpolicyagent.org/" rel="nofollow">Open Policy Agent</a></li>
</ul>
<h2><a id="user-content-scenario-5" class="anchor" aria-hidden="true" href="#scenario-5"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Scenario 5</h2>
<p>Prevent attacks originating via traffic listening or intercepting activity or request forgery (MITM).</p>
<h3><a id="user-content-details-4" class="anchor" aria-hidden="true" href="#details-4"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Details</h3>
<p>We need security at Service level to prevent attacks via <em>Man in the Middle mechanisms</em>(MITM)</p>
<h3><a id="user-content-mitigation-1-use-service-mesh-like-istio-with-envoy" class="anchor" aria-hidden="true" href="#mitigation-1-use-service-mesh-like-istio-with-envoy"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Mitigation 1: Use service mesh like Istio (with Envoy)</h3>
<ul>
<li>Acting as a proxy between services (enforces Service level policies)</li>
<li>E-to-E encryption (Encrypts traffic)</li>
<li>Rolling certs (reduces the window of access in case of malicious access)</li>
<li>Policy managed via central server.</li>
</ul>
<p>Other bits</p>
<ul>
<li>Use Kube-bench for ascertaining issues in cluster or actions pertaining to changes.</li>
</ul>
<p><a href="https://github.com/aquasecurity/kube-bench">kube-bench</a> is a Go application that checks whether Kubernetes is deployed securely by running the checks documented in the <a href="https://www.cisecurity.org/benchmark/kubernetes/" rel="nofollow">CIS Kubernetes Benchmark</a></p>
